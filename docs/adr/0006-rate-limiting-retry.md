# ADR-0006: Rate limiting и retry-стратегия

## Статус
Принято

## Контекст
Публичное API hh.ru (ADR-0003) не имеет выделенных лимитов для
авторизованных клиентов и может возвращать временные ошибки (5xx, таймауты,
троттлинг) при интенсивных запросах — а объём запросов большой: до
20 профессий × (1 запрос списка + до 100 запросов деталей).

## Решение
- Пауза 400мс между последовательными HTTP-запросами.
- Таймаут запроса — 10 секунд.
- При ошибке — до 3 повторов с паузой 2 секунды между ними.
- После исчерпания повторов — пропустить вакансию/профессию и залогировать,
  не прерывая весь сбор.

## Обоснование
- Фиксированная пауза проще экспоненциального бэкоффа и предсказуема по
  времени выполнения (можно оценить ETA для прогресс-бара).
- Пропуск после retry вместо остановки всего процесса — сбор одной
  профессии не должен блокировать остальные 19.

## Последствия
- Полный прогон детерминированно долгий: ~20 профессий × 100 вакансий ×
  400мс ≈ 13+ минут минимум, без учёта retry — учтено в UX (progress ETA,
  раздел 6 ТЗ).
- Единичные сетевые сбои не портят весь результат, но могут занижать
  итоговые счётчики навыков (пропущенные вакансии не учитываются) — это
  фиксируется в логе и итоговой статистике «Ошибок/повторов».
