# Архитектура

## Обзор

SkillMap — однопроцессное консольное приложение без внешних сервисов
(БД, очередей, серверной части). Вся логика умещается в один бинарник,
взаимодействующий с двумя внешними ресурсами: публичным API hh.ru и
локальной файловой системой (кэш, итоговый `.xlsx`).

## Поток данных

```
┌──────────┐   город (текст)   ┌────────────┐
│  Пользо- │ ────────────────> │            │
│  ватель  │ <──────────────── │   main.go  │  диалог, оркестрация
└──────────┘   выбор/прогресс  └─────┬──────┘
                                     │
                    ┌────────────────┼─────────────────┐
                    ▼                ▼                 ▼
              ┌───────────┐   ┌────────────┐    ┌─────────────┐
              │  api.go   │   │  cache.go  │    │ progress.go │
              │ HTTP hh.ru│   │ read/write │    │  \r-вывод   │
              └─────┬─────┘   │  JSON      │    └─────────────┘
                    │         └─────┬──────┘
                    ▼               ▼
             api.hh.ru        cache_<город>.json
                    │
                    ▼ (агрегированные счётчики навыков)
              ┌───────────┐
              │  excel.go │
              └─────┬─────┘
                    ▼
           <город>_навыки.xlsx
```

## Модули и зона ответственности

| Модуль | Отвечает за | Не отвечает за |
|---|---|---|
| `main.go` | CLI-диалог (город, кэш, старт/стоп), оркестрация вызовов остальных модулей | HTTP-запросы, формат кэша, стили Excel |
| `api.go` | HTTP-клиент к hh.ru: поиск города, список вакансий, детали вакансии, retry/rate-limit | Хранение результатов, отображение прогресса |
| `cache.go` | Сериализация/десериализация `cache_<город>.json`, восстановление прогресса | Сетевые запросы |
| `progress.go` | Рендер прогресс-бара и статус-блока в терминал | Бизнес-логика сбора |
| `excel.go` | Сборка агрегированной таблицы, стилизация, запись `.xlsx` | Сбор данных из API |

Разделение по файлам соответствует однонаправленному потоку данных: `main.go`
дирижирует, `api.go` поставляет сырые данные, `cache.go` их сохраняет между
запусками, `progress.go` показывает прогресс, `excel.go` формирует
финальный артефакт. Ни один модуль не импортирует `main`, что позволяет
тестировать `api.go`/`cache.go`/`excel.go` изолированно.

## Жизненный цикл одного запуска

1. Баннер → ввод города → резолв через `/suggests/areas` (`api.go`).
2. Проверка `cache_<город>.json` (`cache.go`) → продолжить или начать заново.
3. Цикл по 20 профессиям:
   - `api.go`: список вакансий → детали → `key_skills` → счётчики.
   - `progress.go`: обновление статус-блока после каждой вакансии.
   - `cache.go`: запись прогресса после каждой завершённой профессии.
4. По завершении цикла — `excel.go`: агрегация всех профессий в одну
   таблицу, сортировка, стилизация, запись `<город>_навыки.xlsx`.
5. Итоговый экран статистики (`main.go` + `progress.go`).

## Ключевые решения

Обоснования конкретных технических выборов (Go, excelize, формат кэша,
retry-стратегия и т.д.) вынесены в [ADR](adr/) — здесь описано только
"как устроено", там — "почему выбрано именно так".
